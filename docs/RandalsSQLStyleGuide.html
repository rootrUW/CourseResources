SQL Style Guide

Randal's Preferred SQL Formatting Guidelines

These formatting guidelines represent the conventions that have consistently proven the most efficient and maintainable throughout my work with multiple organizations over several decades. While SQL style standards can vary from one company to another, using a unified style in this course will help ensure clarity and consistency across all student work.

Goals of this Style Guide

Using this style guide provides the following benefits:

  • Ensures a consistent SQL style across all assignments
  • Reduces mistakes caused by inconsistent formatting
  • Makes your SQL easier for others to read and understand
  • Reduces debugging time caused by formatting issues
  • Helps AI tools produce better results by giving clear formatting expectations

Note: Please use this style guide for all SQL you write in this course.

1. Casing and Formatting Basics

1.1 Indentation

  • Use 2 spaces per indentation level.
  • Do not use tabs. (These can appear differently on other computers, making the format hard to read.)

Example: Basic SELECT with indentation

Select
  c.CustomerId,
  c.CustomerFirstName,
  c.CustomerLastName
From dbo.Customers As c
Where c.IsActive = 1;

1.2 Keyword Casing

Use Title Case for SQL keywords:

Select, From, Where, Group By, Order By, Inner Join, Left Join, Case, End, Begin, Go

1.3 Identifier Casing

  • Use PascalCase (also known as TitleCasing) for all tables and columns.
  • Avoid abbreviations unless universally understood.
  • Good: Customers, CustomerOrders, CustomerFirstName, EnrollmentStatus
  • Avoid: CustTbl, EnrStat
  • Use camelCase (first word lower-case, following words capitalized) for indexes, constraints, views, functions, and procedures.
  • Good: ciCustomers, nciCustomersLastName, pkCustomers, uqCustomerEmailAddress, vCustomers, fCustomersByLastName(), pInsCustomer, pUpdCustomer, pDelCustomer
  • Avoid: names_with_underscores, RandomALLCAPSNames

Note: Avoid underscores in names since they can look like spaces in some UIs.

2. Naming Conventions

2.1 Tables

  • Use plural nouns for table names: dbo.Customers, dbo.Orders.
  • Always specify the schema (namespace).

Example: Table definition

Create Table dbo.Customers (
  CustomerId int Not Null,
  CustomerFirstName varchar(50) Not Null,
  CustomerLastName varchar(50) Not Null,
  CreatedDate datetime Not Null,
  IsActive bit Not Null,
  Constraint pkCustomers Primary Key (CustomerId)
);

2.2 Column Names

  • Use singular nouns for column names: CustomerId, OrderId.
  • Use descriptive names that describe the stored data.
    • CustomerFirstName (instead of just FirstName)
    • CreatedDate (instead of just Date)
  • Use standard suffixes:
    • Id for identifiers (CustomerId, OrderId)
    • Date for dates (OrderDate, CreatedDate)
  • Metadata column names do not need to begin with the table name (CreatedDate instead of CustomerCreatedDate).
  • Booleans begin with Is (IsActive, IsDeleted).

2.3 Data Types

Use lower-case to reduce visual noise and keep the focus on structure and readability.

  • Use the built-in SQL Server data types appropriately: int for integers, varchar(number) or nvarchar(number) for variable-length strings, datetime for dates and times, bit for boolean values, decimal(precision,scale) for precise decimal numbers, etc.

Note: Use nvarchar when Unicode support is required; otherwise, varchar is acceptable.

2.4 Keys and Constraints

  • Primary key: pkTableName (for example, pkCustomers).
  • Foreign key: fkSingleParentPluralChildren (for example, fkCustomerOrders for Orders referencing Customers).

2.5 Indexes

  • Use ciTable for clustered indexes (example: ciOrders).
  • Use nciTableColumn for non-clustered indexes (example: nciOrdersOrderDate).

2.6 Views

  • Base view (closest abstraction of a table): vCustomers
  • Reporting view: vSalesSummary

Example: Base view

Create or Alter View dbo.vCustomers
As
Select
  c.CustomerId,
  c.CustomerFirstName,
  c.CustomerLastName,
  c.CreatedDate,
  c.IsActive
From dbo.Customers As c;

3. Query Formatting

3.1 Layout

  • One clause per line.
  • One column per line in non-trivial queries.
  • Multiple columns per line in trivial queries are OK.
  • Align Join and On blocks.

Example: Multi-table query

Select
  o.OrderId,
  o.OrderDate,
  c.CustomerId,
  c.CustomerFirstName,
  c.CustomerLastName
From dbo.Orders As o
Inner Join dbo.Customers As c
  On c.CustomerId = o.CustomerId
Where o.OrderDate >= '2025-01-01'
Order By
  o.OrderDate Desc;

3.2 Avoid Select *

Always list columns explicitly for non-trivial queries.

Tip: Using Select * is OK for quick testing and trivial queries.

Example: Use seldom

Select * From dbo.Customers;

Example: Use often

Select
  CustomerId,
  CustomerFirstName,
  CustomerLastName,
  CreatedDate
From dbo.Customers;

4. Batches, Semicolons, and Go

  • Terminate every SQL statement with ;.
  • Use Go to separate batches in scripts.

Example: Batches

Create Table dbo.Temp (
  Id int Not Null
);
Go

Insert Into dbo.Temp (Id)
Values (1);
Go

Remember: The purpose of the Go keyword is to inform SQL Server which statements should be run as a single "batch" of code by both grouping and separating sets of one or more statements.

5. Comments and Script Headers

5.1 Script Header

Example: Standard script header

--*************************************************************************--
-- Title: ScriptName
-- Author: YourName
-- Desc: Description of script purpose
-- Change Log: When,Who,What
--   2030-01-01,RRoot,Created Script
--*************************************************************************--

5.2 Commenting

Use comments to explain why something is written the way it is, not just what it does.

Example: Why comments

-- Only include active customers in the report.
Where c.IsActive = 1;

6. DDL Standards (Create/Alter Table)

Use one column definition per line and group constraints at the end. Keep the syntax "tight" instead of trying to align types and nullability.

Example: DDL style

Create Table dbo.Customers (
  CustomerId int Not Null,
  CustomerFirstName varchar(50) Not Null,
  CustomerLastName varchar(50) Not Null,
  CustomerEmailAddress varchar(255) Null,
  CreatedDate datetime Not Null Default (GetDate()),
  IsActive bit Not Null Default (1),
  Constraint pkCustomers Primary Key (CustomerId),
  Constraint uqCustomerEmailAddress Unique (CustomerEmailAddress)
);

7. Transactions and Error Handling

Example: Transaction with TRY/CATCH

Begin Try
  Begin Tran;
    Update dbo.Customers
    Set IsActive = 0
    Where CustomerId = 5;
  Commit Tran;
End Try
Begin Catch
  If (@@TranCount > 0) Rollback Tran;

  Declare @ErrorMessage varchar(4000) = Error_Message();
  Raiserror(@ErrorMessage, 16, 1);
End Catch;

Using AI Tools with This Style Guide

Students: You are encouraged to use AI tools to help with formatting and improving the readability of your SQL code. However, you are still responsible for understanding the code you submit and verifying that the logic is correct.

Tip: You can paste this style guide into any AI tool and ask it to format your SQL according to these standards. A recommended workflow is:

  1. Tell the AI your intent.

    For example:

    "Please reformat my SQL code using the SQL Style Guide provided below. Do not change my logic. I will paste the code when you are ready."
  2. Paste this paragraph of the style guide into the AI's input so it knows the rules to follow.
    3. Please reformat my SQL code using the following style rules: use Title Case for SQL keywords; use PascalCase for table and column names; use camelCase for indexes, constraints, views, functions, and stored procedures; use lowercase for all data types; indent with 2 spaces (no tabs); place one clause per line and one column per line in non-trivial queries; avoid Select * except for quick testing; align Join and On blocks; keep DDL syntax tight with one column per line and constraints grouped at the end; end all statements with semicolons; use Go to separate batches. Do not change my query logic—only formatting and readability.
  3. Paste your SQL code when the AI is ready.
  4. Review and edit the AI's output as needed to ensure accuracy and correctness.
