Overview

This demo introduces two related ideas. First, you will see how to create a database and run simple SQL statements such as Create, Insert, Select, Update, and Delete. Second, you will learn the basic rules of normalization, which help you organize data into tables that are easier to maintain, query, and protect from common data problems.

As you move through the examples, pay attention to how a poor design can still work at first, but creates problems as the database grows. Normalization is about building tables in a way that avoids those problems.

Data Design and Atomic Values

A relational database stores data as rows and columns, but the process starts by breaking large pieces of information into smaller, meaningful parts. These smaller pieces are often called atomic values. For example, a full name and phone number stored together as one string are harder to search, sort, and validate than separate first name, last name, and phone number values.

Example: Instead of storing Bob Smith 206.444.1111 as one value, a better design breaks it into FirstName, LastName, and PhoneNumber.

Once the data is broken into useful columns, rows can be grouped into tables, and tables can be related to one another using keys.

Rule: A Table Should Focus on One Subject

A table should usually describe one main thing, such as customers, products, classes, or orders. When unrelated subjects are mixed together in one table, duplicate data and update problems are more likely.

Example of a weak design

Create Table dbo.CustomersProducts (
  CustomerName varchar(50) Null,
  CustomerPhone varchar(50) Null,
  ProductName varchar(50) Null,
  Price varchar(50) Null
);
Go

This works, but it mixes customer data and product data in one table. A better design splits the data into separate tables.

Improved design

Create Table dbo.Customers (
  CustomerName varchar(50) Null,
  CustomerPhone varchar(50) Null
);
Go

Create Table dbo.Products (
  ProductName varchar(50) Null,
  ProductPrice varchar(50) Null
);
Go

Rule: Every Table Needs a Primary Key

A primary key helps identify one row from another. Without a primary key, it is harder to update, delete, or relate specific rows accurately.

Create Table dbo.Customers (
  CustomerId int Not Null,
  CustomerName varchar(50) Null,
  CustomerPhone varchar(50) Null,
  Constraint pkCustomers Primary Key (CustomerId)
);
Go

Create Table dbo.Products (
  ProductId int Not Null,
  ProductName varchar(50) Null,
  ProductPrice varchar(50) Null,
  Constraint pkProducts Primary Key (ProductId)
);
Go

Primary keys are one of the most important ideas in relational design because they give each row a stable identity.

First Normal Form: No Multi-Part Fields

First Normal Form means each column should contain one value, not several pieces of information packed together.

Example of a multi-part field

Create Table dbo.Customers (
  CustomerId int Not Null,
  CustomerName varchar(50),
  CustomerPhone varchar(50),
  CustomerAddress varchar(200),
  Constraint pkCustomers Primary Key (CustomerId)
);
Go

The address column above contains street, city, state, and zip code together. A better design separates those values.

Improved design

Create Table dbo.Customers (
  CustomerId int Not Null,
  FirstName varchar(50),
  LastName varchar(50),
  Phone varchar(50),
  Address varchar(100),
  City varchar(50),
  State char(2),
  Zip char(5),
  Constraint pkCustomers Primary Key (CustomerId)
);
Go

First Normal Form: No Multi-Valued Fields

A column should also avoid storing multiple values inside one row, such as a comma-separated list of product IDs or quantities.

Weak design

Create Table dbo.BadDesignSales (
  SalesId int Not Null,
  CustomerId int,
  ProductId varchar(50) Null,
  Qty varchar(50) Null,
  Constraint pkBadDesignSales Primary Key (SalesId)
);
Go

Storing multiple product IDs and quantities in one row makes the data hard to query and validate. A better design uses a parent sales table and a child line item table.

Improved design

Create Table dbo.Sales (
  SalesId int Not Null,
  CustomerId int,
  Constraint pkSales Primary Key (SalesId)
);
Go

Create Table dbo.SalesLineItems (
  SalesId int,
  LineItemId int,
  ProductId int,
  Qty int,
  Constraint pkSalesLineItems Primary Key (SalesId, LineItemId)
);
Go

Second Normal Form

Second Normal Form matters when a table has a composite primary key. In that case, every non-key column should depend on the full key, not just part of it.

For example, if SalesLineItems uses SalesId and LineItemId as a composite key, then a column such as SalesDate does not belong there if it depends only on SalesId. That date should be stored in the Sales table instead.

Create Table dbo.Sales (
  SalesId int Not Null,
  CustomerId int,
  SalesDate date,
  Constraint pkSales Primary Key (SalesId)
);
Go

Third Normal Form

Third Normal Form means non-key columns should depend on the primary key, not on some other non-key column. For example, if a sales row stores CustomerId, then customer name and phone number should normally come from the customer table rather than being repeated in the sales table.

Repeating customer details in the sales table can lead to update problems because the same customer data may need to be changed in many places.

Common Relationship Types

Relational databases often use three main relationship patterns.

Relationship Type Description Example
One-to-One One row in one table matches one row in another table. Employees and EmployeeSensitiveData
One-to-Many One row in one table matches many rows in another table. Categories and Products
Many-to-Many Many rows in one table match many rows in another table through a bridge table. Sales and Products through SalesLineItems

One-to-One Example

One-to-one related tables are sometimes used for performance, privacy, or security. For example, an employee table might store public information, while a separate related table stores salary and SSN data.

Create Table dbo.Employees (
  EmployeeId int Not Null,
  FirstName varchar(50),
  LastName varchar(50),
  Phone varchar(50),
  HireDate datetime,
  Constraint pkEmployees Primary Key (EmployeeId)
);
Go

Create Table dbo.EmployeeSensitiveData (
  EmployeeId int Not Null,
  SSNumber varchar(50),
  Salary money,
  Address varchar(100),
  City varchar(50),
  State char(2),
  Zip char(5),
  Constraint pkEmployeeSensitiveData Primary Key (EmployeeId)
);
Go

One-to-Many Example

One-to-many relationships are very common. A category can have many products, but each product belongs to one category.

Create Table dbo.Categories (
  CategoryId int Not Null,
  CategoryName varchar(50),
  Description varchar(200),
  Constraint pkCategories Primary Key (CategoryId)
);
Go

Create Table dbo.Products (
  ProductId int Not Null,
  ProductName varchar(50),
  Price varchar(50),
  CategoryId int,
  Constraint pkProducts Primary Key (ProductId)
);

Summary

In this demo you saw how database design moves from raw values to well-structured tables. You also saw why normalization matters. Good design keeps each table focused on one subject, gives each row a key, avoids multi-part and multi-valued fields, and places columns in the table where they logically belong. These ideas are foundational for everything else you will do in SQL.
