Overview

This module introduces four common database object types that help make databases more reliable, faster, and easier to use: constraints, indexes, views, and stored procedures. These objects are not just advanced features. They are part of everyday database design and development.

In this demo you will see how constraints protect data, how indexes improve performance for common searches, how views display data through saved queries, and how stored procedures save reusable SQL code.

1) Constraints

Constraints are database rules that help protect the quality of the data. SQL Server supports several types of data integrity, including entity integrity, domain integrity, referential integrity, and user-defined integrity.

Think of constraints as built-in rules that the database enforces automatically. This means the database itself can help stop invalid data before it is saved.

Create a demo table with several constraints

Create Table dbo.Demo (
  Col1 int Not Null,
  Col2 int,
  Col3 int,
  Col4 int,
  Col5 int,
  Col6 int Not Null,
  Constraint pkDemo Primary Key (Col1),
  Constraint uqDemoCol2 Unique (Col2),
  Constraint ckDemoCol3 Check (Col3 > 0),
  Constraint fkDemoCol4 Foreign Key (Col4) References dbo.Demo (Col2),
  Constraint dfDemoCol5 Default (0) For Col5
);
Go

Look at metadata about constraints

Exec sp_Help Demo;
Go

Exec sp_HelpConstraint Demo;
Go

SQL Server also stores metadata about constraints in system views.

Select *
From sys.default_constraints;

Select *
From sys.check_constraints;

Select *
From sys.key_constraints;

Select *
From sys.foreign_keys;
Go

Default constraints

A default constraint provides a value automatically when no value is supplied.

Create Table dbo.Products (
  ProductId int Identity(1,1) Not Null,
  ProductName nvarchar(40) Not Null,
  SupplierId int Null,
  UnitPrice money Null,
  UnitsInStock smallint Null,
  Constraint pkProducts Primary Key (ProductId),
  Constraint dfProductsUnitPrice Default (0) For UnitPrice,
  Constraint dfProductsUnitsInStock Default (0) For UnitsInStock
);
Go

Primary key, unique, foreign key, and check constraints

Create Table dbo.Suppliers (
  SupplierId int Not Null,
  ContactId int Not Null,
  Constraint pkSuppliers Primary Key Clustered (SupplierId)
);
Go

Alter Table dbo.Suppliers
Add Constraint uqSuppliersContactId
  Unique Nonclustered (ContactId);
Go

Alter Table dbo.Products
Add Constraint fkProductsSuppliers
  Foreign Key (SupplierId)
  References dbo.Suppliers (SupplierId)
  On Update Cascade
  On Delete Cascade;
Go

Alter Table dbo.Products
Add Constraint ckProductsUnitPrice Check (UnitPrice >= 0);
Go

Try this: Insert rows that break each rule and watch SQL Server reject them. This is one of the easiest ways to understand what each constraint protects.

2) Indexes

Indexes help SQL Server find rows faster. They do this by organizing or copying key values in a way that makes common searches more efficient.

Heap table example

When a table is first created without a clustered index, it is stored as a heap.

Create Table dbo.PhoneList (
  Id int,
  Name varchar(50),
  Extension char(5)
);
Go

Insert Into dbo.PhoneList (
  Id,
  Name,
  Extension
)
Values
  (1, 'Bob Smith', '#11'),
  (2, 'Sue Jones', '#12'),
  (3, 'Joe Harris', '#13');
Go

Clustered index

A clustered index changes the physical order used for the table data.

Create Clustered Index ciPhoneListId
On dbo.PhoneList (Id);
Go

Nonclustered index

A nonclustered index stores a separate structure that helps SQL Server search more efficiently for specific columns.

Create Nonclustered Index nciPhoneListName
On dbo.PhoneList (Name);
Go

Test common queries

Select *
From dbo.PhoneList
Order By Id;

Select *
From dbo.PhoneList
Where Name = 'Joe Harris';

Select
  Name
From dbo.PhoneList;
Go

Turn on the Execution Plan in SQL Server Management Studio or Azure Data Studio to see when SQL Server uses the table directly and when it uses an index.

3) Views

A view is a saved Select statement. Views do not normally store data. Instead, they display data from one or more tables through a reusable query.

Views are useful because they can simplify query writing and provide an abstraction layer between tables and applications.

Use Northwind;
Go

Create View vCustomerOrders
As
Select
  Orders.OrderId,
  Customers.CompanyName,
  Customers.ContactName
From Orders
Inner Join Customers
  On Orders.CustomerId = Customers.CustomerId;
Go

Use the view like a table

Select *
From vCustomerOrders;
Go

See the definition of the view

Select *
From syscomments
Where id = Object_Id('vCustomerOrders');
Go

Exec sp_HelpText vCustomerOrders;
Go

Try this: Compare the full join query to the simple Select * from the view. This helps show why views are useful as a reusable abstraction layer.

4) Stored Procedures

A stored procedure is saved SQL code that can be executed as a unit. Unlike a view, a stored procedure can contain multiple statements and can also accept parameters.

Use Northwind;
Go

Create Procedure pGetEmployeeData
As
Begin
  Select
    FirstName,
    LastName,
    HireDate
  From Employees;
End;
Go

Execute the stored procedure

Exec pGetEmployeeData;
Go

See the stored procedure text

Select *
From syscomments
Where id = Object_Id('pGetEmployeeData');
Go

Exec sp_HelpText 'pGetEmployeeData';
Go

Stored procedures are often used for transaction processing, repeated business operations, and security. They let you keep important processing logic inside the database.

Comparing the Object Types

Object Main Purpose Example Use
Constraint Protect data quality Prevent negative prices or duplicate keys
Index Improve query performance Speed up searches by name or id
View Display data through a saved query Show customer and order data together
Stored Procedure Save executable SQL code Run a standard report or process transaction logic

Lab Ideas

After working through the demos, practice by creating your own objects in a lab database.

  1. Create three related tables such as Classes, Students, and StudentsClasses.
  2. Add primary keys, foreign keys, and a check constraint.
  3. Create one or two indexes to improve likely searches.
  4. Create a view for each table.
  5. Create a stored procedure that returns rows from one table or view.
